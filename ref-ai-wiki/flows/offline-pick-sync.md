# Flow: Offline Pick Sync

## Purpose

Document the complete lifecycle of picks made while the user has no network connection, from the moment the user taps a pick button through queue storage, reconnect detection, deduplication, and the Supabase flush that achieves eventual consistency.

---

## Relevant Source Files

- `src/repositories/picks.ts` — `toggle()`, `flushOfflineQueue()`
- `src/lib/db.ts` — `enqueueOfflinePick()`, `loadOfflineQueue()`, `removeFromOfflineQueue()`, `saveUserPick()`, `removeUserPick()`
- `src/App.tsx` — `PickSync` component (flush on init + 'online' event)
- `src/components/SyncToast.tsx` — "Synced N items" toast
- `src/components/OfflineBanner.tsx` — Offline status indicator
- `src/hooks/useOfflinePendingBandIds.ts` — Exposes pending band IDs for PendingChip rendering

---

## Trigger

The user taps the pick/unpick button for a band while `navigator.onLine === false`, OR the Supabase write fails while online (network error, 5XX, etc.).

---

## Object Store Structure

IndexedDB has **13 object stores**. Picks live in two of them:

| Store | Key | Purpose |
|-------|-----|---------|
| `user_picks` | `[user_id, band_id]` | Current confirmed picks (primary data) |
| `offline_picks` | `string (uuid)` | Queue of unsynced pick operations |

Each offline queue entry has this shape:

```typescript
type OfflinePickOp = {
  id: string;           // e.g., "alice-uuid:band42:<crypto-uuid>"
  user_id: string;
  band_id: string;
  action: 'add' | 'remove';
  created_at: string;   // ISO 8601 — used for dedup ordering
};
```

The `id` is generated by `offlinePickId(userId, bandId)` which concatenates `userId:bandId:<crypto-uuid>` so queue entries are always unique even for the same band toggled multiple times.

## Happy Path (Online Pick)

When the user is online, the flow completes entirely within `toggle()`:

```
User taps pick button
         │
         ▼
picksRepository.toggle(userId, bandId, currentlyPicked)
         │
         ├─ [unpick path] removeUserPick(userId, bandId)   [IndexedDB]
         │  [pick path]   saveUserPick(pick)               [IndexedDB]
         │  → emitPicksChanged()                           [window event]
         │  → UI re-renders immediately (optimistic)
         │
         ├─ navigator.onLine === true, so:
         │
         │  [unpick] supabase.from('user_picks').delete().eq(...)
         │  [pick]   supabase.from('user_picks').upsert(pick)
         │
         │  if error → queuePick(userId, bandId, action, now)
         │           → enqueueOfflinePick(op) in offline_picks store
         │
         └─ Done — no queue entry if Supabase succeeded
```

**Timeline (online)**:
```
T=0ms  — User taps
T=1ms  — IndexedDB write → PICKS_CHANGED_EVENT → UI updates
T=50ms — Supabase upsert begins (async, doesn't block UI)
T=150ms— Supabase confirms OK → no queue entry
```

## Offline Behavior

When `navigator.onLine === false`, the toggle exits immediately after queuing:

```
User taps pick button (offline)
         │
         ▼
picksRepository.toggle(userId, bandId, currentlyPicked)
         │
         ├─ saveUserPick(pick)              [IndexedDB — immediate]
         │  emitPicksChanged()              [window event]
         │  → UI shows pick ✓ (optimistic)
         │
         ├─ navigator.onLine === false:
         │  await queuePick(userId, bandId, 'add', now)
         │  → db.put('offline_picks', {
         │      id: "alice:band42:abc123...",
         │      user_id: "alice",
         │      band_id: "band42",
         │      action: "add",
         │      created_at: "2026-07-29T18:50:00Z"
         │    })
         │
         └─ Return early (no Supabase call)
```

**User experience**: Pick looks immediately confirmed. A `PendingChip` appears on the BandCard (via `useOfflinePendingBandIds`). The `OfflineBanner` shows "🚫 Offline — Changes will sync when back online".

## Deduplication: The keepLast Semantics

### The Problem

A user offline can toggle a band many times before reconnecting. Storing every operation naively means:

```
Queue (5 ops for alice + band42):
  T=10:00  add
  T=10:05  remove
  T=10:10  add
  T=10:15  remove
  T=10:20  add
```

5 Supabase calls on reconnect — but the final state is just **add** (1 call needed).

### The Solution

`flushOfflineQueue()` groups operations by `(user_id, band_id)` and keeps only the **last** (chronologically):

```typescript
// Step 1: Sort queue oldest-first
const queue = (await loadOfflineQueue()).sort((a, b) =>
  a.created_at.localeCompare(b.created_at)
);

// Step 2: Group by (user_id, band_id), tracking last action
const groups = new Map<string, { all: Op[]; last: Op }>();
for (const op of queue) {
  const key = `${op.user_id}:${op.band_id}`;
  const g = groups.get(key);
  if (g) {
    g.all.push(op);   // accumulate all for batch deletion
    g.last = op;      // overwrite to keep latest
  } else {
    groups.set(key, { all: [op], last: op });
  }
}

// Step 3: Sync only the last action per group
for (const { all, last } of groups.values()) {
  const { error } = last.action === 'add'
    ? await supabase.from('user_picks').upsert({ user_id, band_id, created_at })
    : await supabase.from('user_picks').delete().eq('user_id', ...).eq('band_id', ...);

  // Step 4: Remove all queue entries atomically only on success
  if (!error) {
    await Promise.all(all.map(op => removeFromOfflineQueue(op.id)));
    flushed += all.length;
  }
}
```

**Key properties**:
- **keepLast by timestamp**: The `created_at` sort ensures the loop always processes oldest→newest, making `g.last = op` always point to the most-recent action.
- **Atomic removal**: All queue entries for a group are deleted only when the Supabase call succeeds. If Supabase fails, all entries remain for the next retry.
- **Idempotent**: Calling `flushOfflineQueue()` twice is safe — the second call sees an empty or partial queue.

## Worked Example: 5 Toggles → 1 Sync Call

**Scenario**: User picks Slipknot offline, changes mind 4 times, decides to keep the pick.

```
T=0:10  User picks Slipknot (offline)
  → offline_picks: [{id: "u1:s1:A", action: "add",    created_at: "10:10"}]
  → user_picks:    [{user_id: "u1", band_id: "s1"}]      ← UI: ✓ picked

T=0:15  User unpicks Slipknot (still offline)
  → offline_picks: [{id: "u1:s1:A", action: "add",    created_at: "10:10"},
                    {id: "u1:s1:B", action: "remove",  created_at: "10:15"}]
  → user_picks:    []                                     ← UI: unpicked

T=0:20  User picks again
  → offline_picks: [{..A add 10:10}, {..B remove 10:15}, {..C add 10:20}]
  → user_picks:    [{user_id: "u1", band_id: "s1"}]      ← UI: ✓ picked

T=0:25  User unpicks again
  → offline_picks: [{..A add 10:10}, {..B remove 10:15}, {..C add 10:20}, {..D remove 10:25}]
  → user_picks:    []                                     ← UI: unpicked

T=0:30  User picks (final)
  → offline_picks: [{..A add 10:10}, ..., {..E add 10:30}]
  → user_picks:    [{user_id: "u1", band_id: "s1"}]      ← UI: ✓ picked

─────────────── Network returns at T=1:00 ───────────────

'online' event → PickSync → flushOfflineQueue()

Sort by created_at → A(10:10) B(10:15) C(10:20) D(10:25) E(10:30)

Group by (u1:s1):
  all = [A, B, C, D, E]
  last = E  (action='add')

Sync: supabase.from('user_picks').upsert({user_id:'u1', band_id:'s1', ...E})
  → SUCCESS

Remove: removeFromOfflineQueue(A.id, B.id, C.id, D.id, E.id)

flushed = 5
SyncToast fires: "✓ Synced 5 items"

Final state:
  offline_picks: []  (empty)
  user_picks:    [{user_id:'u1', band_id:'s1'}]  (picked ✓)
  Supabase:      user_picks INSERT confirmed
```

## Sync Behavior (Reconnect)

### Trigger

The `PickSync` component in `App.tsx` listens for two triggers:

1. **On login** (userId becomes available) — flushes any residual queue from previous session
2. **On 'online' event** — flushes queue whenever browser detects network restored

```typescript
function PickSync() {
  const { session } = useAuth();
  const userId = session?.user?.id;

  useEffect(() => {
    if (!userId) return;

    async function syncNow() {
      const [picksFlushed, presenceFlushed] = await Promise.all([
        picksRepository.flushOfflineQueue(),
        presenceRepository.flushOfflineQueue(),
      ]);
      if (picksFlushed + presenceFlushed > 0) emitSyncComplete();

      await Promise.all([
        picksRepository.syncCrewFromRemote(),
        usersRepository.syncCrew(),
        presenceRepository.syncCrewFromRemote(),
      ]);
    }

    syncNow().catch(() => {});

    window.addEventListener('online', () => syncNow().catch(() => {}));
    return () => window.removeEventListener('online', ...);
  }, [userId]);
}
```

**After flush**:
- `emitSyncComplete()` fires → `SyncToast` renders "✓ Synced N items"
- `syncCrewFromRemote()` fetches all crew picks → overwrites IndexedDB → `PICKS_CHANGED_EVENT`
- All components (attendance counts, /my-picks, /popular) re-render with fresh data

## Error Recovery

| Error | Behavior | Recovery |
|-------|----------|----------|
| Supabase returns error | Queue entry stays | Retried on next 'online' event |
| Supabase network timeout | Same as error | Retried automatically |
| Validation error (e.g., band_id FK violation) | Queue entry stays | Never resolves unless band exists |
| IndexedDB write fails | `enqueueOfflinePick` throws | Unhandled; pick is stored in user_picks but never queued — may desync |
| User logs out before sync | Queue persists in IndexedDB | Flushed on next login (PickSync fires on userId change) |

**Philosophy**: Never fail the user's operation. If we can't sync, queue it. If we can't queue it, the user still sees the optimistic state in IndexedDB.

---

## Realtime Updates After Reconnect

After the offline queue is flushed, `syncCrewFromRemote()` does a full pull:

```typescript
async function syncCrewFromRemote(): Promise<void> {
  const { data } = await supabase.from('user_picks').select('*');
  await replaceUserPicks(data as UserPick[]);
  // replaceUserPicks calls emitPicksChanged()
}
```

This means all attendance counts are atomically refreshed. Any picks made by other crew members while this user was offline will appear immediately after reconnect.

## Edge Cases

### 1. User Goes Offline Mid-Write

If the Supabase `upsert()` is in-flight when network drops:

- The Supabase call will error (fetch rejected)
- `picks.ts` catches the error and calls `queuePick()`
- IndexedDB already has the optimistic write from before the Supabase call
- Result: pick appears confirmed in UI AND is in the queue; on reconnect, `upsert` is called again — idempotent

### 2. Multiple Sessions (Same User, Two Devices)

Both devices can pick the same band offline simultaneously. On reconnect:
- Both call `supabase.upsert({user_id, band_id})` 
- `upsert` has an `ON CONFLICT DO UPDATE` semantic → idempotent, no duplicate rows
- Both devices end up consistent

### 3. Queue Has Entry But Band No Longer Exists

Rare: lineup changes + cache version bump clears all local data. The queue is cleared by `wipeAllLocalData()` which deletes all IndexedDB stores including `offline_picks`. Net result: the offline pick is lost (acceptable for a metal festival app with an immutable lineup).

### 4. Flush Partially Succeeds

If the queue has 10 ops across 5 bands and Supabase fails on band 3:

- Bands 1–2: ops removed from queue ✓
- Band 3: ops remain in queue (will retry on next 'online')
- Bands 4–5: continue processing (failure is per-group, not global abort)

The loop is not wrapped in a transaction — partial success is intentional to maximize throughput.

### 5. Very Large Queue (100+ ops)

No queue size limit. All ops are flushed in a single `flushOfflineQueue()` call. Since dedup runs first, 100 ops for 20 bands reduces to 20 Supabase calls. Performance is acceptable for ~20 users and 78 bands.

---

## Known Limitations

1. **No dedup at enqueue time**: Dedup runs only at flush time. The queue can grow to N entries per band. This is deliberate — simpler code, and N is small in practice.

2. **No exponential backoff**: If Supabase is down, `flushOfflineQueue()` is called on every 'online' event (which can fire multiple times as signal fluctuates). This can cause a burst of failed calls.

3. **No per-entry TTL**: Queue entries from a session 3 days ago will still attempt to sync on next login. Usually harmless (upsert is idempotent) but can cause unexpected behavior if the user's picks were externally changed.

4. **`navigator.onLine` is not reliable**: The browser can report `true` even when network is degraded. The Supabase call timeout/error will still trigger queueing, so this is a latency issue, not a correctness issue.

---

## Future Improvements

- **Dedup at enqueue time**: Before adding to the queue, check if an opposite action is already queued for the same (user_id, band_id) and cancel them out.
- **Exponential backoff**: Delay retry on 'online' event if last flush partially failed.
- **Queue size warning**: Alert the user (or godlike) if the queue exceeds 50 entries.
- **Sync progress UI**: Instead of just a toast, show "Syncing 12/20 items..." for large batches.
